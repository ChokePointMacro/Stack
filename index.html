Three Claude surfaces, three jobs. The trick is sending each task to the one built for it instead of forcing everything through chat. Below: what each is for, a one-line rule, and a routing table for the work you actually do.

Think · Research · Draft

Claude Chat

This window. No repo or file-system access on its own — it reasons, researches, and drafts things you carry elsewhere.

  • The analytical thesis behind a ChokePointMacro report
  • Architecture & sequencing calls before you touch code
  • Prose, captions, outlines, reading lists
  • One-off SQL, regex, snippets, prompt design
Agent · Local files · Office

Claude Cowork

Desktop agent. Point it at a folder plus connected apps (Excel, PowerPoint, Chrome, Gmail, Drive) and it runs multi-step work.

  • FP&A models & LLC workbooks in real .xlsx
  • Investor / brand decks in PowerPoint
  • Running the PDF pipeline over a data folder
  • Email / calendar / Asana triage
Agent · Your repo

Claude Code

Terminal / IDE agent working directly in a Git repo — reads files, runs commands, ships changes with a review diff.

  • The Armed Capital app, auth, and Stripe wiring
  • Supabase schema & migrations
  • Vercel config, the CI gate, cron jobs
  • Trading bot & the ReportLab pipeline code
Quick rule. Does it touch the repo? → Code. Does it act on local files, Office, or connected apps? → Cowork. Is it thinking, research, drafting, or a one-off? → Chat.

Goal: your Windows laptop, Mac laptop, and Windows desktop are interchangeable clients, not three separate setups. The rule that makes that true — nothing important lives only on one machine. Code goes to GitHub, assets go to Drive, secrets come from one source, and every machine pulls the same state.

Next session · on the Mac

Clerk is off the stack, so Supabase Auth is now the only auth layer. If any app code still calls Clerk, it has to move over before sign-in works. Open the repo in Claude Code and paste:

Migrate this Next.js app from Clerk to Supabase Auth. • Audit all Clerk usage: @clerk/nextjs, ClerkProvider, middleware, auth() / currentUser(), <SignIn> / <SignUp>. • Replace with Supabase Auth via @supabase/ssr — server + client helpers and middleware session refresh. • Rebuild sign-in / sign-up / sign-out UI and protected routes. • Update RLS policies that reference auth.uid(); migrate user IDs. • Remove Clerk packages + env vars; update .env.example. • Keep Clerk code until the Supabase flow is verified end-to-end, then run the build + a manual auth smoke test before committing.

Two-commitments rule still applies — let this be the one build task you finish before opening the next.

1 · One source of truth

Code → GitHub. Commit and push before you switch machines; pull when you sit down at the next one. Uncommitted work is the only thing that can't follow you, so don't let it pile up.

Assets → Google Drive. Finished PDFs, decks, and research go to Google Drive, never a local Downloads folder. Supabase, Vercel, and Stripe are already cloud — same data on every machine by default.

Secrets → one source. Never commit .env. Pull environment variables on each machine with vercel env pull, or keep keys in 1Password/Bitwarden so all three machines share one copy. PowerShell and WSL don't share env vars, so set them in the right place on each ($PROFILE vs ~/.bashrc).

2 · The three Claude surfaces, per machine

Claude Chat all 3 + phone

Zero setup. History and Projects sync through your account everywhere. Put ChokePointMacro and Armed Capital in Projects so the context follows you between machines instead of being re-explained.

Claude Cowork

macOS ✓Windows x64 ✓ARM64 Win — partial

Install the desktop app on the Mac and both Windows machines (x64). Sign in once each, connect the same Connectors, and set per-folder instructions so you're not re-prompting. Because Cowork acts on a local folder, keep that working folder synced (Drive or the repo) so all three see identical files. Start in Ask mode until you trust a given workflow.

Claude Code

macOS ✓Windows native ✓WSL2 optional

Native install on each machine — no Node required for the app itself. Commit a CLAUDE.md at the repo root so your project instructions travel with the code to every machine. You can continue a desk session from your phone via remote control at claude.ai/code.

# Windows (PowerShell) — laptop + desktop irm https://claude.ai/install.ps1 | iex winget install Git.Git # gives Claude Code the Bash tool # macOS brew install --cask claude-code
Windows tip: native is fine for the Next.js app. If the Python pipeline or bash scripts misbehave, run Claude Code inside WSL2 on that machine for full Unix parity — just don't expect PowerShell and WSL to share your env vars.

3 · Environment parity — the Windows ↔ Mac friction

Node — pin one version. Use nvm on Mac and nvm-windows or fnm on Windows, then add a .nvmrc to the repo so nvm use picks the same version on every machine.

Python pipeline — reproduce it. A venv + requirements.txt covers the libraries, but your pdf2image preview step needs poppler: brew install poppler on Mac vs. installing the poppler binaries and adding them to PATH on Windows. This is the single most likely thing to break the pipeline across OSes.

Line endings. Add a .gitattributes with * text=auto so Windows CRLF and Mac LF stop fighting and producing noisy diffs.

CLIs everywhere. Install GitHub CLI, Vercel CLI, and Supabase CLI on each machine — winget on Windows, brew on Mac. Avoid hardcoded OS paths in scripts.

Three repeat offenders: poppler missing on Windows, CRLF/LF churn, and env vars not crossing the PowerShell↔WSL boundary. Fix these once per machine and the rest is interchangeable.

4 · Per-machine setup checklist

Run the same list on each of the three machines once; after that, day-to-day is just clone-pull-push.

  • Install Git, a Node version manager, and Python
  • git clone the repo → nvm use → npm install
  • vercel env pull (or pull keys from your password manager)
  • python -m venv → pip install -r requirements.txt → install poppler
  • Install Claude Code (native) and sign in
  • Open Cowork, sign in, connect the same Connectors, set folder instructions
  • Sign into Chat; confirm your Projects appear
  • Run claude doctor to verify the install
Which machine for what? Any of them — pick by where you're sitting. The Windows desktop earns the long Code and Cowork sessions (more RAM, sustained runs); the laptops are for mobility; your phone is for remote-continuing a Code session. Keep them synced by the rules above and it genuinely doesn't matter which one you open.
